# Mirage Integration Plan

Date: 2026-05-27

## 1. Goal

Turn the existing Mirage implementation from a partially wired architecture into a verified experimental MVP.

The immediate target is not production support. The target is a small, reproducible proof that Qdrant can build a Mirage-backed dense segment and serve search requests through the existing Qdrant search path.

## 2. Current Starting Point

Already implemented:

- `Indexes::Mirage(MirageConfig)`.
- `VectorIndexEnum::Mirage(MirageIndex)`.
- `MirageIndex::build(...)` and `MirageIndex::open(...)`.
- Rust-native Layer 0 refinement builder.
- HNSW-compatible graph persistence.
- HNSW runtime delegation for search.
- Constructor support if segment config already contains Mirage.
- Basic `ef_construct` and mismatch compatibility hooks.

Main missing point:

- A normal collection/optimizer flow still produces `Indexes::Hnsw`, not `Indexes::Mirage`.

## 3. Target MVP Definition

MVP constraints:

- Single-node local Qdrant.
- Dense vector only.
- Single vector name.
- Float32 first.
- No payload filter for the first smoke test.
- No sparse vector.
- No multi-vector.
- No payload subgraph.
- No inline storage.
- No GPU build path.
- No distributed/sharded validation.
- No public API change initially.

MVP success means:

1. A test or local run can force indexed dense segments to use Mirage.
2. `VectorIndexEnum::Mirage` is instantiated during segment build/open.
3. A Qdrant search request reaches `MirageIndex::search(...)`.
4. Search returns valid Qdrant point IDs.
5. Results are compared against exact search on a small dataset.
6. Build time and recall are measured against HNSW.

## 4. Phase 0: Add an Explicit Experimental Trigger

Add a hidden local switch first. Do not expose public REST/gRPC schema yet.

Suggested env var:

```text
QDRANT_EXPERIMENTAL_MIRAGE=1
```

Behavior:

- If the env var is disabled, Qdrant behaves exactly as today.
- If enabled, indexed dense vectors that would normally become HNSW become Mirage.

Implementation sketch:

```rust
fn experimental_mirage_enabled() -> bool {
    std::env::var("QDRANT_EXPERIMENTAL_MIRAGE")
        .map(|v| v == "1" || v.eq_ignore_ascii_case("true"))
        .unwrap_or(false)
}
```

Primary insertion point:

- `lib/shard/src/optimizers/segment_optimizer.rs`

Current behavior:

```rust
config.index = Indexes::Hnsw(vector_cfg.hnsw_config);
```

Experimental behavior:

```rust
if experimental_mirage_enabled() {
    config.index = Indexes::Mirage(MirageConfig::from_hnsw(vector_cfg.hnsw_config));
} else {
    config.index = Indexes::Hnsw(vector_cfg.hnsw_config);
}
```

Needed helper:

```rust
impl MirageConfig {
    pub fn from_hnsw(hnsw: HnswConfig) -> Self {
        MirageConfig {
            m: hnsw.m,
            ef_construct: 1024.max(hnsw.ef_construct),
            full_scan_threshold: hnsw.full_scan_threshold,
            max_indexing_threads: hnsw.max_indexing_threads,
            on_disk: hnsw.on_disk,
            payload_m: None,
            ..Default::default()
        }
    }
}
```

Decision:

- For P0, set `payload_m: None` even if HNSW had payload graph config.
- Do not silently claim filter-optimized payload subgraphs are supported.

## 5. Phase 1: Guardrails for Experimental Scope

Add explicit validation so unsupported combinations fail early or clearly fall back.

Recommended P0 behavior:

- `payload_m.is_some()`:
  - reject or force `None` under the experimental env var with a warning.
- `inline_storage`:
  - not present on `MirageConfig`; document as unsupported.
- multi-vector:
  - reject for P0 unless current HNSW delegation is explicitly tested.
- sparse:
  - unaffected.
- filter:
  - for the first segment-level smoke test, test `filter = None`.
  - Do not claim filter support until there is a dedicated filtered recall/correctness test.
- quantization:
  - build path currently accepts `quantized_vectors`; do not claim support until tested.

Preferred rule:

```text
MVP supports no-filter dense float32 search. Other paths are experimental and unvalidated.
```

## 6. Phase 2: Segment-Level Test

Write a focused segment integration test before running full REST tests.

Test intent:

- Build a small segment with dense vectors.
- Configure vector data as `Indexes::Mirage`.
- Build the segment index.
- Assert the vector index is `VectorIndexEnum::Mirage`.
- Run search with no filter.
- Compare top-k with exact/plain search.

Suggested dataset:

- 1,000 to 5,000 vectors.
- 32 or 64 dimensions.
- Euclidean first.
- deterministic RNG seed.

Assertions:

- Build succeeds.
- Search returns `top` results.
- Returned internal IDs are valid and not deleted.
- Recall@10 against exact search is above a loose MVP threshold.
- Telemetry index name is `mirage`.

Why segment-level first:

- It avoids collection/shard optimizer timing.
- It proves the index abstraction and graph construction are correct.
- It is faster to debug than REST-level behavior.

## 7. Phase 3: REST Smoke Test

After segment-level proof, run a local Qdrant smoke test.

Run configuration:

```bash
QDRANT_EXPERIMENTAL_MIRAGE=1 ./target/debug/qdrant
```

Collection:

- single dense vector,
- one shard,
- no replication,
- indexing threshold low enough to trigger indexing,
- no payload filters.

Test flow:

1. Create collection.
2. Upsert vectors.
3. Wait for optimization/indexing.
4. Search with `filter = null`.
5. Verify results are returned.
6. Inspect logs or telemetry to confirm `mirage` index was used.

Acceptance:

- No crash.
- Search API returns valid external point IDs.
- Results roughly match exact search.
- Segment files include `mirage_config.json`.

## 8. Phase 4: Correctness Benchmark

Before performance claims, build a repeatable correctness benchmark.

Compare:

- exact/plain search,
- HNSW,
- Mirage.

Metrics:

- recall@10,
- recall@100,
- build time,
- query latency p50/p95,
- indexed vector count,
- graph file size,
- memory during build if easy to collect.

Datasets:

- Synthetic random vectors for quick CI-like checks.
- One realistic embedding dataset for meaningful recall.

Initial acceptance target:

- Mirage build time is measurably lower than HNSW on the same dataset.
- Recall at comparable `hnsw_ef` is not catastrophically worse.
- Query latency is comparable because runtime search is HNSW.

Do not overclaim:

- A small synthetic benchmark does not prove production performance.
- Filtered and quantized paths require separate measurements.

## 9. Phase 5: Performance Cleanup

Known performance concern:

- The current build path creates `FilteredScorer` repeatedly during pairwise scoring.

This may erase Mirage's expected construction advantage.

Cleanup tasks:

1. Add per-thread scorer reuse.
2. Avoid scorer allocation inside tight pairwise loops.
3. Profile `build_layer0(...)`.
4. Count distance computations and scorer constructions.
5. Compare before/after build time.

Likely implementation options:

- Rayon-local scorer cache.
- Thread-local scorer holder keyed by entry point.
- A lower-level pairwise scoring API if Qdrant already exposes one safely.

Validation:

- Same recall/correctness as before.
- Build time improves.

## 10. Phase 6: Persistence and Reopen Test

Because current Mirage writes graph files and config, persistence needs explicit testing.

Test:

1. Build Mirage segment.
2. Search once.
3. Drop/reopen segment.
4. Search again.
5. Compare results.
6. Assert `files()` includes `mirage_config.json`.
7. Assert `indexed_vector_count()` is stable.

This is required before discussing snapshot or production use.

## 11. Phase 7: Filter and Payload Strategy

Qdrant's README emphasizes extended filtering, so this cannot be ignored long-term.

But Mirage P0 should not claim filter-optimized behavior.

Plan:

1. First validate no-filter search.
2. Then validate simple payload filter correctness through HNSW runtime delegation.
3. Measure whether filtered search falls back to plain scan or uses graph traversal.
4. Keep `payload_m` unsupported until there is a Mirage-specific payload subgraph strategy.

Decision point:

- If HNSW runtime delegation handles filter correctness without Mirage-specific payload graphs, document it as functionally supported but not performance-optimized.
- If payload subgraphs are required for performance parity, design that separately.

## 12. Phase 8: Public API Decision

Do not expose public `type: mirage` until the experimental path is proven.

Public API work would require:

- REST schema changes.
- OpenAPI generation.
- gRPC proto changes if exposed through gRPC.
- client compatibility review.
- config migration/reload tests.
- documentation.

Recommended sequence:

1. Hidden env var.
2. Internal tests and benchmarks.
3. Feature flag or internal config.
4. Public API only after support matrix is clear.

## 13. Risk Register

High risk:

- Build performance due to repeated scorer creation.
- Incorrect graph connectivity if Layer 0 injection and upper-layer entry-point logic interact badly.
- Mismatch between Mirage-specific config and optimizer rebuild logic.
- Overstating filter/quantization support before tests exist.

Medium risk:

- Persistence/reopen edge cases.
- Deleted vectors and id tracker semantics.
- Multi-vector behavior.
- On-disk graph cache behavior.

Low risk:

- Adding enum match arms is already mostly done.
- Delegating runtime search to HNSW is architecturally sound if the graph format is valid.

## 14. Recommended Immediate Task List

Priority order:

1. Add `QDRANT_EXPERIMENTAL_MIRAGE` helper.
2. Add `MirageConfig::from_hnsw(...)`.
3. Modify indexed dense optimizer path to emit `Indexes::Mirage` under the env var.
4. Add clear logging when Mirage override is active.
5. Add a segment-level Mirage search test.
6. Add a reopen test for `mirage_config.json` and HNSW-compatible files.
7. Run `cargo test -p segment mirage`.
8. Run one local REST smoke test.
9. Add a small benchmark script or bench target.
10. Profile scorer construction in `build_main_graph(...)`.

## 15. Suggested Milestone Definition

Milestone A: Triggerable MVP

- Env var triggers Mirage.
- Segment-level search test passes.
- REST smoke test returns valid results.

Milestone B: Correctness Baseline

- Exact vs Mirage recall benchmark exists.
- HNSW vs Mirage build-time comparison exists.
- Reopen test passes.

Milestone C: Performance Baseline

- Scorer reuse implemented or consciously deferred with measured cost.
- Build-time improvement is shown on at least one non-trivial dataset.

Milestone D: Support Matrix

- Explicit yes/no/experimental matrix for:
  - filters,
  - quantization,
  - on-disk,
  - snapshots,
  - updates/deletes,
  - multi-vector,
  - distributed deployment.

Only after Milestone D should public API exposure be considered.
