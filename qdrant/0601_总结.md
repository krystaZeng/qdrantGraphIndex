# 0601 总结：Mirage Index 当前实现、进度与风险

## 1. 总体结论

当前状态不能严格表述为“已经完成可正式对外调用的 Qdrant Mirage API”。

更准确的判断是：

- segment/internal 层已经可以构建、打开并搜索 Mirage index。
- collection public config 入口已经初步接入。
- optimizer resolver 已能识别 explicit Mirage，并与 experimental Mirage flag 区分。
- C++ golden 对齐已经覆盖 Layer0 refinement、Layer0 injection、persisted GraphLayers。
- 普通 search path 在已构建 Mirage segment 上可以命中 Mirage。
- 但 REST create collection 端到端、OpenAPI、gRPC、update collection/config diff、长期兼容策略仍未完成。

因此：

```text
MVP/internal experimental：接近可用，但仍有若干必须收紧的风险点。
正式 public API：未完成。
```

## 3. MVP 进度

按“内部实验 MVP”标准，当前进度约为：

```text
75% - 80%
```

已经具备：

- Mirage index 本体可 build/open/search。
- 可以通过 segment constructor 得到 `VectorIndexEnum::Mirage`。
- C++ golden 对齐已覆盖关键路径。
- no-filter search recall 可解释，当前 recall@100 = `0.959`。
- collection public config 已有初步 API model。
- explicit Mirage 不 silent fallback。
- threshold 语义已实现：显式 Mirage 不绕过 Qdrant indexing threshold。

MVP 仍需完成：

- REST 端到端测试。
- 文档 JSON 示例与实际 `MirageConfig` 字段对齐。
- explicit Mirage + optimizer storage policy 的边界处理。
- `m = 0` 等会污染实验的配置拒绝。
- ConfigMismatchOptimizer 对 explicit Mirage rebuild 的测试。

MVP 剩余工作量估算：

```text
20-35 单人小时
```

## 4. 产品化进度

按“可以作为 Qdrant 正式 public API 发布”标准，当前进度约为：

```text
40% - 50%
```

主要缺口：

- gRPC proto / conversion 未支持。
- OpenAPI snapshot 未更新。
- update collection/config diff 未设计。
- 当前直接暴露 segment 内部 `MirageConfig`，长期 public API 最好拆出 public `MirageIndexParams` / `MirageConfigDiff`。
- 参数上界、内存风险、`m=0` 等保护不足。
- REST/gRPC/SDK round-trip 未验证。
- filtered search 未来策略未产品化，目前只是 reject。
- on-disk、quantization、payload-aware subgraph 都还没支持。

产品化剩余工作量估算：

```text
80-140 单人小时
```

## 5. 当前能否调用 Qdrant Mirage API

分层回答：

1. 直接在 segment/internal 层调用：可以。
   - `MirageIndex::build/open/search` 已经能工作。
   - segment integration 测试通过。

2. 通过 Qdrant 普通 search API，在已构建 Mirage segment 上搜索：理论路径可行。
   - Mirage 是 vector index backend，不是新 search endpoint。
   - 搜索仍走普通 search API。

3. 通过正式 REST create collection API 配置 Mirage 并稳定使用：还不能算完成。
   - public model 已初步接入 `index.type = mirage`。
   - 但缺 REST E2E、OpenAPI、gRPC、update collection/config diff、产品化 validation 收紧。

准确表述：

```text
现在已经能在 Qdrant 内部/segment 层像 HNSW 一样构建并搜索 Mirage。
但还不能宣称已经完成可正式对外调用的 Qdrant Mirage API。
```

## 6. Code Review 发现的问题

### 6.1 High：文档/API 示例当前不能按代码反序列化

问题：

- `0601_mirage_api_design.md` 的 JSON 示例没有 `full_scan_threshold`。
- 但当前 public API 直接复用 `segment::types::MirageConfig`。
- `MirageConfig.full_scan_threshold` 没有 serde default，是必填字段。
- 测试里已经需要补 `full_scan_threshold: 10000` 才能反序列化。

影响：

- 用户按文档示例创建 collection 会失败。
- 这是 public API 和文档不一致。

更稳妥方案：

- 短期：文档示例补 `full_scan_threshold`。
- 中期：给 `MirageConfig.full_scan_threshold` 加 default。
- 产品化：拆 public `MirageIndexParams` / `MirageConfigDiff`，在 collection -> optimizer 阶段 resolve 成内部 `MirageConfig`。

### 6.2 High：explicit Mirage 可能被 optimizer storage policy 改成 P0 不支持的 storage

问题：

- `apply_dense_storage_policy` 可能把 storage 改成 `InRamMmap` / `Mmap`。
- explicit Mirage resolver 当前直接返回 `Indexes::Mirage(cfg)`。
- 这会绕过 P0 guard。
- 后续 segment constructor 会拒绝 mmap/on-disk storage，导致优化失败。

影响：

- 不会 silent fallback，但会导致 optimizer build 失败。
- 可能造成用户认为 Mirage API 不稳定。

更稳妥方案：

- explicit Mirage 时 storage policy 应保持 P0 支持的 `InRamChunkedMmap` / `Memory`。
- 或让 resolver 返回 `Result<Indexes>`，在不支持 storage 时显式报错。
- 需要补 explicit Mirage + memmap threshold / feature flag 的测试。

### 6.3 Medium：`m = 0` 会让 explicit Mirage 退化成 Plain-like Mirage，污染实验指标

问题：

- `MirageConfig.m` 没有最小值校验。
- `MirageIndex::build` 中 `m == 0` 会跳过 main graph。
- 用户显式配置 Mirage，但实际行为接近 Plain。

影响：

- benchmark 会被污染。
- telemetry / index kind 看起来是 Mirage，但实际不是正常 Mirage。

更稳妥方案：

- explicit Mirage P0 validation 拒绝 `m == 0`。
- 如果未来要支持 `m = 0`，应定义为“禁用索引”，不要返回 `VectorIndexEnum::Mirage`。

### 6.4 Medium：gRPC conversion 会静默丢弃 `index`

问题：

- grpc -> collection 时 `index` 被置为 `None`。
- collection -> grpc 时 `index` 被忽略。
- 当前 P0 如果定位 internal/REST-only 可以接受，但正式 public API 不可接受。

影响：

- gRPC 创建、导出、复制、SDK round-trip 会丢 Mirage 配置。
- 可能导致 HNSW/Plain 混入 Mirage 实验。

更稳妥方案：

- 短期：加注释和测试，明确 P0 gRPC 不支持 Mirage。
- 正式发布前：补 gRPC proto / conversion / SDK round-trip。

### 6.5 Medium：public Mirage 参数缺少上界，存在内存/构建时间风险

问题：

- `m` 无范围。
- `s/r/iter/num_reverse_edges` 只有下界。
- `m0 = m.saturating_mul(2)` 避免整数溢出，但不能避免巨大图和大 pool 带来的内存压力。

影响：

- 用户可提交极端参数导致构建耗时或内存爆炸。
- 对多租户/生产环境不安全。

更稳妥方案：

- P0 加保守上界，例如：
  - `m <= 128`
  - `s <= 1024`
  - `r <= 64`
  - `iter <= 1024`
  - `num_reverse_edges <= 4096`
- 具体上界可根据 benchmark 和内存模型再收紧。

## 7. 逻辑正确性评估

整体主链路是正确的：

```text
VectorParams.index
  -> DenseVectorOptimizerConfig.mirage_config
  -> resolve_desired_dense_index
  -> Indexes::Mirage
  -> build_vector_index
  -> VectorIndexEnum::Mirage
```

threshold 语义也是正确的：

```text
如果 threshold 未达到，显式 Mirage 也仍然返回 Indexes::Plain。
index.type = mirage 只表示“需要建索引时使用 Mirage backend”。
```

explicit Mirage 与 experimental flag 的优先级也合理：

1. threshold 未达到：Plain。
2. explicit Mirage：Mirage。
3. experimental flag：eligible 时 Mirage，否则 HNSW。
4. default：HNSW。

需要注意的是：

- explicit Mirage 当前需要在 resolver 或 storage policy 阶段继续收紧 P0 storage guard。
- ConfigMismatchOptimizer 必须始终复用同一 resolver，避免 HNSW/Mirage target 不一致。

## 8. 边界条件评估

已覆盖：

- distance 只能是 `Euclid`。
- datatype 只能是 Float32。
- quantization reject。
- multivector reject。
- vector on_disk reject。
- Mirage index on_disk reject。
- payload_m reject。
- runtime filtered search reject。
- threshold 未达到时 Plain。

仍需补：

- `m == 0` reject。
- `m/s/r/iter/num_reverse_edges` 上界。
- `full_scan_threshold` 默认值或 public params resolve。
- explicit Mirage + memmap threshold / single file mmap feature flag。
- gRPC round-trip 丢配置。
- update collection diff 与 existing Mirage config 的兼容。

## 9. 是否会污染实验指标

目前有污染风险：

1. `m = 0` explicit Mirage 会生成 Plain-like Mirage。
2. gRPC round-trip 会丢 `index`，可能退回 HNSW/Plain。
3. REST 文档示例缺 `full_scan_threshold`，用户可能改用 legacy HNSW config 绕过 Mirage。
4. ConfigMismatchOptimizer 如果缺 explicit Mirage rebuild 测试，未来改动可能导致 HNSW/Mirage 混跑。
5. optimizer storage policy 如果导致 Mirage build 失败，用户可能手动切换或 fallback，影响 benchmark 可复现性。

## 10. 内存、类型与溢出风险

当前没有明显 unsafe 内存问题。

已做得较稳的点：

- `m0 = m.saturating_mul(2)` 避免整数溢出。
- `payload_m0 = payload_m.map(|v| v.saturating_mul(2))` 也避免整数溢出。
- `full_scan_threshold * BYTES_IN_KB` 使用 saturating/checked 逻辑。

仍有风险：

- 巨大的 `m/s/num_reverse_edges` 会导致图边、候选池和 refinement 内存压力。
- 巨大的 `r/iter` 会导致构建时间爆炸。
- `m = 0` 语义不适合作为 explicit Mirage public API。
- 直接暴露 internal `MirageConfig` 会把内部字段和默认值泄漏给 public API。

## 11. 需要补的测试

必须补：

1. 文档 JSON 示例反序列化测试
   - 使用 `0601_mirage_api_design.md` 中的示例。
   - 验证是否需要 `full_scan_threshold`。

2. explicit Mirage + `m = 0`
   - 应在 collection validation 阶段 reject。

3. explicit Mirage + storage policy
   - memmap threshold 触发后不应构建失败。
   - single file mmap feature flag 打开时不应把 explicit Mirage 变成 P0 不支持 storage。

4. ConfigMismatchOptimizer rebuild
   - existing HNSW segment + explicit Mirage target => 必须触发 rebuild。
   - existing Mirage + changed Mirage config => 必须触发 rebuild。
   - existing Mirage + same config => 不 rebuild。

5. gRPC round-trip
   - 当前若 REST-only P0，应明确测试“gRPC 不支持 Mirage index config，会丢弃/拒绝”。
   - 正式 API 前应改为 round-trip 保留。

6. update collection
   - existing Mirage 时提交 legacy `hnsw_config` diff 应拒绝。
   - Mirage config diff 后续支持时应触发 rebuild。

7. filtered search runtime reject
   - collection-level / segment-level 都应覆盖。
   - 已有 segment-level guard，但需要 REST/search 层 E2E 覆盖。

## 12. 更稳妥的实现路线

短期 MVP 修正：

1. 文档和测试补 `full_scan_threshold`，或给 `MirageConfig.full_scan_threshold` 加 default。
2. collection validation 拒绝 `m == 0`。
3. 给 Mirage 参数加保守上界。
4. explicit Mirage 时禁止 storage policy 改成 P0 不支持的 mmap/on-disk。
5. 补 ConfigMismatchOptimizer explicit Mirage rebuild 测试。
6. 补 REST create collection -> insert -> optimize -> search E2E。

产品化路线：

1. 拆 public `MirageIndexParams` / `MirageConfigDiff`。
2. collection 层 resolve 成 internal `segment::types::MirageConfig`。
3. 支持 update collection/config diff。
4. 同步 gRPC proto / conversion。
5. 更新 OpenAPI snapshot。
6. 补 REST/gRPC/SDK round-trip。
7. 明确 filtered search、quantization、on-disk、payload-aware subgraph 的长期策略。

建议按这个顺序推进：

1. 修复 P0 高风险问题

   - `m = 0` reject。

2. 补防污染测试

4. 决定 public API 分层
   - P0 继续复用 `MirageConfig`。
   - 或进入产品化前拆 public params/diff。
