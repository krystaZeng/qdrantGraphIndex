# Mirage 今日计划

日期：2026-05-28

## 1. 已实现部分

目前 Mirage 已经不只是概念代码，而是已经接入到了 Qdrant 的索引抽象层里。

已经完成的部分包括：

1. 已新增 `Indexes::Mirage(MirageConfig)`，说明 segment 配置层已经能表达 Mirage 索引类型。
2. 已新增 `VectorIndexEnum::Mirage(MirageIndex)`，说明运行时索引枚举已经能承载 Mirage 实例。
3. 已实现 `MirageIndex::build(...)` 和 `MirageIndex::open(...)`，具备构建和加载 Mirage 索引的基本能力。
4. 已完成 Rust 原生的 Layer 0 refinement builder。
5. 已实现与 HNSW 兼容的 graph 持久化。
6. 搜索阶段目前通过 HNSW runtime delegation 复用已有搜索路径。
7. 如果 segment config 中已经显式配置 Mirage，构造流程可以创建 Mirage 索引。
8. 已有基础的 `ef_construct` 兼容逻辑和 mismatch compatibility hook。

当前最关键的缺口是：普通 collection / optimizer 流程仍然默认生成 `Indexes::Hnsw`，还不会自动切换到 `Indexes::Mirage`。

## 2. 今天目标

今天的核心目标是把 Mirage 从“手动配置可用”推进到“实验开关可触发，并能被测试验证”。

优先级如下。

## 3. 任务一：增加实验开关

在 `lib/shard/src/optimizers/segment_optimizer.rs` 附近增加隐藏环境变量：

```text
QDRANT_EXPERIMENTAL_MIRAGE=1
```

预期行为：

- 默认不影响 Qdrant 原有行为。
- 开启后，符合 P0 边界的 dense vector indexed segment 改为构建 Mirage。
- 如果开启后遇到 multi-vector、非 Float32、quantization、on-disk vector storage 等未验证组合，继续回退到 HNSW 并记录日志。
- 暂时不暴露 REST / gRPC 公共 API。

建议实现一个集中判断函数：

```rust
fn experimental_mirage_enabled() -> bool {
    std::env::var("QDRANT_EXPERIMENTAL_MIRAGE")
        .map(|v| v == "1" || v.eq_ignore_ascii_case("true"))
        .unwrap_or(false)
}
```

## 4. 任务二：增加 `MirageConfig::from_hnsw(...)`

把现有 HNSW 配置转换成 Mirage 配置，先复用关键字段：

- `m`
- `ef_construct`
- `full_scan_threshold`
- `max_indexing_threads`

P0 阶段先固定：

```rust
payload_m: None
```

不要声称已经支持 payload subgraph / filter 优化。

`on_disk` P0 先不做：

- 不继承 `hnsw.on_disk`。
- 不在 optimizer 层开放 Mirage on-disk 行为。
- 不声明 Mirage on-disk build / restore / search 已验证。
- `from_hnsw()` 显式固定为 `Some(false)`，表示 P0 只做 in-memory Mirage，不作为 on-disk 支持。

建议实现形式：

```rust
impl MirageConfig {
    pub fn from_hnsw(hnsw: HnswConfig) -> Self {
        MirageConfig {
            m: hnsw.m,
            ef_construct: hnsw.ef_construct.max(1024),
            full_scan_threshold: hnsw.full_scan_threshold,
            max_indexing_threads: hnsw.max_indexing_threads,

            // P0: Mirage is in-memory only. Do not inherit HNSW on_disk yet.
            on_disk: Some(false),

            // P0: no payload-aware subgraph / filter optimization.
            payload_m: None,
            ..Default::default()
        }
    }
}
```

## 5. 任务三：增加实验范围保护

先明确 MVP 范围：

- dense vector only
- float32 first
- no filter
- no sparse vector
- no multi-vector
- no quantization claim
- no distributed validation

说明：

- `single vector name` 暂时作为实验建议，不作为硬性代码 guard。
- Qdrant 是 per vector name 构建 index；如果多个 named dense vector 各自都满足 P0 条件，可以分别构建 Mirage。
- 真正需要防御的是 multi-vector per point，也就是 `multivector_config.is_some()`。

如果遇到不支持路径，优先选择清晰失败或日志提示，不要静默假装支持。

推荐规则：

```text
MVP supports no-filter dense float32 search. Other paths are experimental and unvalidated.
```

任务三按三层 guard 实现。

### 5.1 optimizer 自动切换路径

这一层保护 `QDRANT_EXPERIMENTAL_MIRAGE=1` 触发的自动 optimizer 路径。

P0 规则：

- 仅 `multivector_config = None` 时允许切 Mirage。
- 仅 `datatype = None` 或 `datatype = Float32` 时允许切 Mirage。
- 仅未启用 quantization 时允许切 Mirage。
- collection 显式 `on_disk = true` 或 memmap threshold 已触发时，不切 Mirage，继续使用 HNSW。

处理策略：

- 满足 P0 条件：`Indexes::Mirage(MirageConfig::from_hnsw(...))`。
- 不满足 P0 条件：回退 `Indexes::Hnsw(...)`，并记录 warning。

`datatype` 语义必须保持一致：

- 允许 `datatype = None`，因为它表示默认 Float32。
- 允许 `datatype = Some(VectorStorageDatatype::Float32)`。
- 拒绝 `Float16`、`Uint8` 和后续新增的其它非 Float32 类型。

### 5.2 segment constructor 手动 Mirage 路径

optimizer guard 只能保护自动切换路径。如果用户或测试直接构造：

```rust
Indexes::Mirage(...)
```

仍然可能绕过 optimizer。因此需要在 segment constructor 的 Mirage build/open 分支前增加 P0 校验。

建议增加私有函数：

```rust
fn validate_mirage_p0_vector_config(vector_config: &VectorDataConfig) -> OperationResult<()> {
    // reject multivector_config.is_some()
    // reject datatype other than None / Float32
    // reject quantization_config.is_some()
    // reject storage_type.is_on_disk()
    // reject MirageConfig.on_disk == Some(true)
    // reject MirageConfig.payload_m.is_some()
}
```

调用位置：

- `open_vector_index(...)` 的 `Indexes::Mirage(mirage_config)` 分支。
- `build_vector_index(...)` 的 `Indexes::Mirage(mirage_config)` 分支。

处理策略：

- 手动 Mirage 配置不满足 P0 条件时返回 `OperationError::ValidationError`。
- 不要自动降级成 HNSW，因为用户显式要求了 Mirage，静默降级会让测试和 benchmark 产生误判。

### 5.3 Mirage runtime 兜底防御

Mirage 自身还需要低层防御，避免未来新增入口绕过 segment constructor。

建议在 `MirageIndex::build(...)` 和 `MirageIndex::open(...)` 内部检查：

- `quantized_vectors.borrow().is_some()` 时 reject。
- `vector_storage.borrow().try_multi_vector_config().is_some()` 时 reject。
- vector storage 是 sparse / multi / byte / half / mmap / on-disk 相关变体时 reject。
- `mirage_config.on_disk == Some(true)` 时 reject。
- `mirage_config.payload_m.is_some()` 时 reject。

处理策略：

- 返回 `OperationError::ValidationError`。
- 不使用 `service_error`，因为这些是实验能力边界，不是系统损坏或内部异常。

### 5.4 filter search guard

P0 明确不声明 filter search 支持，因此 `MirageIndex::search(...)` 需要先拦截 filter：

```rust
if filter.is_some() {
    return Err(OperationError::ValidationError {
        description: "Mirage P0 supports no-filter search only".to_string(),
    });
}
```

query 类型 guard 可以稍后补。当前优先保证 no-filter dense search 路径可验证。

### 5.5 on-disk 语义

需要区分三类 on-disk：

- vector storage on-disk / mmap：影响向量存储是否 mmap / on-disk。
- `MirageConfig.on_disk`：表示 Mirage index 自身是否 on-disk。
- collection mmap threshold：可能触发向量存储转 mmap。

P0 统一语义：

```text
只要 Mirage 会依赖 mmap/on-disk vector storage，或者 Mirage index 自身声明 on_disk，就不走 Mirage。
```

处理策略：

- optimizer 自动路径：回退 HNSW。
- 手动 Mirage 路径：返回 `ValidationError`。

### 5.6 quantization 三层防御

quantization 配置可以继续在 optimizer 中按原逻辑传递，但 Mirage P0 必须明确不支持 quantized build/search。

防御策略：

1. optimizer 自动路径：
   - `quantization_config.is_some()` 时不自动切 Mirage，回退 HNSW。
2. segment constructor 手动 Mirage 路径：
   - `Indexes::Mirage(...)` 且 `quantization_config.is_some()` 时 reject。
3. Mirage runtime 路径：
   - `quantized_vectors.borrow().is_some()` 时 reject。

这样可以避免 benchmark 或 smoke test 误以为 Mirage 已经支持 quantization。

### 5.7 任务三测试建议

轻量单元测试：

- optimizer env on，但 multi-vector 时回退 HNSW。
- optimizer env on，但 `datatype = Uint8 / Float16` 时回退 HNSW。
- optimizer env on，但 quantization 开启时回退 HNSW。
- optimizer env on，但 collection `on_disk = true` 或 memmap threshold 触发时回退 HNSW。
- 手动 `Indexes::Mirage + multivector_config` 返回 `ValidationError`。
- 手动 `Indexes::Mirage + datatype = Uint8 / Float16` 返回 `ValidationError`。
- 手动 `Indexes::Mirage + quantization_config.is_some()` 返回 `ValidationError`。
- 手动 `Indexes::Mirage + storage_type = Mmap` 返回 `ValidationError`。
- `MirageConfig { on_disk: Some(true) }` 返回 `ValidationError`。
- `MirageConfig { payload_m: Some(_) }` 返回 `ValidationError`。
- `MirageIndex::search(filter = Some(_))` 返回 `ValidationError`。

## 6. 任务四：写 segment-level 测试

先不急着跑完整 REST 流程，优先写一个更稳定的 segment 集成测试。

测试内容：

- 构建 1000 到 5000 条向量。
- 维度 32 或 64。
- 使用 deterministic seed。
- 配置 `Indexes::Mirage`。
- 触发 index build。
- 断言实际索引是 `VectorIndexEnum::Mirage`。
- 执行 no-filter search。
- 和 exact/plain search 对比 recall@10。

今天的最低验收标准：

- Mirage index 能被实例化。
- `MirageIndex::search(...)` 能被打到。
- 返回合法 point id。
- 小数据集 recall 结果可解释。

## 8. Rust 学习安排

今天 Rust 学习建议直接服务于 Mirage 改造，不单独刷泛泛知识。

重点学习内容：

1. `enum` 与模式匹配

   结合 `Indexes::Hnsw` / `Indexes::Mirage` / `VectorIndexEnum` 看 Qdrant 如何做类型分发。

2. 配置转换与 ownership

   重点看 `HnswConfig` 传参方式，是 move、clone 还是引用，避免为了编译到处 `.clone()`。

3. `impl Default` 与结构体更新语法

   重点理解：

   ```rust
   MirageConfig {
       m: hnsw.m,
       ..Default::default()
   }
   ```

4. 环境变量读取

   学会封装一个小函数，例如：

   ```rust
   fn experimental_mirage_enabled() -> bool
   ```

   并理解为什么这种实验开关不应该散落在多个地方。

5. Qdrant 测试结构

   今天重点读 segment-level 测试，不需要一次理解完整 shard / collection / optimizer 全链路。

建议时间分配：

- 30 分钟：读 optimizer 和 index config 相关代码。
- 60 到 90 分钟：实现实验开关和 config 转换。
- 90 分钟：写 segment-level 测试。
- 30 分钟：Rust 复盘，记录今天遇到的 ownership / enum / config 问题。
- 剩余时间：跑测试和修编译错误。
